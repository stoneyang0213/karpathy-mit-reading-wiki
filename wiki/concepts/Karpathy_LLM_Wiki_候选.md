# Karpathy LLM Wiki(外部概念)

> ⚠️ _候选 标记:外部概念,不在 14 本书里,但是项目的架构基石

## 来源
Andrej Karpathy 在 2026 年 4 月发布的 `llm-wiki` gist,介绍他个人用 LLM 维护知识库的工作流。
Karpathy 自己的版本积累到约 100 篇文章 / 40 万字。

## 三层架构

### 1. raw/(原始层,不可变)
- 放 PDF、论文、文章、笔记、代码、图片、截图等原始材料
- LLM 只读,永不修改
- 确保数据真实性和可追溯性

### 2. wiki/(LLM 维护层)
- Markdown 文件的有结构目录
- 摘要页 / 实体页 / 概念页 / 交叉引用
- **完全由 LLM 生成和维护**(人只提供材料)

### 3. claude.md(规则层)
- 定义 LLM 的行为规范
- 包含 ingest / query / lint 流程
- 是知识库的"法律"

## 三个操作

### Ingest(摄入)
- 人把新材料放进 raw/
- LLM 读取、提炼、生成新的 wiki 页面
- 更新交叉引用,写日志

### Query(查询)
- 人提问
- LLM 先查 wiki/ 而不是 raw/(wiki 是"已编译的知识")
- 有价值的新答案**回写**到 wiki/,实现复利增长

### Lint(校验)
- LLM 定期扫描 wiki
- 找矛盾、过时信息、缺失链接
- 提出修复建议

## 关键优势

- **持续编译** vs 传统 RAG 的"实时检索":wiki 是"编译后"的知识,查询时 token 消耗更低
- **主动关联**:LLM 自动发现知识间的潜在关联,形成图谱
- **低维护成本**:人只管喂材料和问问题,LLM 负责组织

## 与 MIT 48 小时法的合体

Karpathy 的架构给出了"**知识存在哪里**"的答案,但没给出"**第一次 ingest 怎么 ingest 得好**"的协议。

MIT 48 小时法恰好填补了这个空缺 —— 它提供了**第一次 ingest 一个陌生领域时该问的 3 个问题**。

合体公式:
```
claude.md ingest 规则 = 对每本新书,按 MIT 3 问生成:
  1. 贡献给 frameworks/ 的 1-3 个共识论点
  2. 贡献给 debates/ 的立场(如有)
  3. 贡献给 tests/ 的候选辨别题
```

这个合体在 2026-04-13 的公开网络上**没人做过**(已 WebSearch 验证)。
这是 stoneyang 项目的原创贡献。
