# Day 8+ 工具候选清单

> **用途**: 存放 14 本书 demo 进行中出现的"想装但现在不装"的工具 / 服务 / 基础设施
> **重新评估时机**: Day 7 文章发布之后,进入 Day 8+ 的迭代期再逐一评估
> **纪律**: demo 进行中(Day 1-7)出现的装新工具冲动 = `feedback_tool_procrastination` 陷阱,默认延后到这个文件里

## 为什么用这个文件

stoneyang 的最大陷阱是"用学习回避输出"。装新工具是这个陷阱的高发形态。
但"想装却不装"的念头如果没地方放,会变成"错过焦虑",反而更扰人。

这个文件解决的就是**错过焦虑** —— 把种子存下来,你不会忘,但你也不去装。
Day 7 demo 发布之后,这个文件里的每一条都会被重新评估,决定装 / 不装 / 删除。

## 候选 #1:Wave Terminal

### 基本信息
- **官网 / 仓库**: https://github.com/wavetermdev/waveterm
- **官网**: https://waveterm.dev
- **License**: Apache-2.0
- **技术栈**: Go (48.8%) + TypeScript (43.1%) + Electron

### 2026-04-14 晚 verification 结果
- ⭐ **19.5k stars**(对比:昨晚的 cmux-win 是 0 stars 全新项目)
- 📦 **2,600 commits**,活跃维护
- 🗓️ 最近发布 **v0.14.4 (2026-03-27)**,3 周前
- 🔒 完备安全文档:SECURITY.md / CoC / CONTRIBUTING

### 为什么候选(stoneyang 想装它的原因)
- 它是一个现代化终端多路复用器,支持 tab / panes / 内嵌 web / AI 集成
- stoneyang 对"升级工作流工具"有天然的兴趣
- 比目前用的 Git Bash 视觉和交互体验更现代

### 2026-04-14 为什么**不**立刻装
- **时机冲突**:Day 5 的第一个动作是写文章第 5 节,不是"探索新终端"
- **成本代价**:装 Wave Terminal 后 stoneyang **必然**会花 30-60 分钟探索它的 tab / panes / 主题 / AI 集成,这 30-60 分钟直接从写作时间里扣
- **启动路径冲突**:刚刚花半小时写的 `start-project.bat` 启动路径(双击 → Ctrl+V → 回车)依赖 Windows 默认终端行为,换终端可能要重新调试
- **工具本身没问题**,问题是**此刻装它这件事是拖延**

## 添加规则

每次 stoneyang 冒出装新工具的冲动,Claude 按 `feedback_tool_procrastination` 默认拦截,然后:
1. 做一次 verification(确认工具是否合法)
2. 给 A / B / C 三选项,C 就是"加到这个文件里"
3. 如果选 C,把候选信息结构化写入本文件
4. 不去实际装它
